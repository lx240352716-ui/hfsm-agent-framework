# fill 阶段规则

> 选参考行 + 填核心值 + 树形确认。参考行候选来自 locate 阶段。

## 参考行选择

locate 阶段的 `locate_result.json` 包含每个模块的 `ref_candidates` 或 `_ref_id`：

- `ref_status = "found"` + 有 `ref_candidates` → 从候选中选最合适的，确定 `_ref_id`
- `ref_status = "found"` + 有 `_ref_id` → 已确定，直接使用
- `ref_status = "not_found"` → **向用户询问参考 ID，不要从零盲填**

## not_found 处理

如果某张表 `ref_status = "not_found"`：
1. 向用户说明"该表未找到合适参考"
2. 请用户提供一个参考 ID
3. 用户提供后，将 ID 写入 `locate_result.json` 对应模块的 `_ref_id`
4. 触发 `retry_locate` 回到 locate 重新验证

## 核心值填写

确定 `_ref_id` 后，填写需求差异值：

### _overrides
明确要改的字段值，格式 `{ 字段英文名: 值 }`：
- 售价 → `currencyInfo: "1,0,100"`
- 限购 → `limitData: "2,5,0,0,0"`

### _note
自然语言补充说明（给 L2 理解用）：
- 什么字段需要替换新 ID
- 名字/描述改成什么

## 数值字段定位

| 字段 | 表 | 参考值来源 |
|------|----|-----------|
| 阵法经验值曲线 | FormationLev.经验值 | 参考同类阵法（席卷之阵等级0-4） |
| 阵法助阵人数 | FormationLev.助阵人数 | 参考同类阵法 |
| buff参数(type=1速度) | BuffActive.p1 | 百分比值（如0.3=+30%） |
| buff参数(type=7坚韧) | BuffActive.p1/p2 | 来自战斗策划模板 |

## 取值规则

| 类型 | 规则 |
|------|------|
| 速度百分比 | p1直接是小数（0.3=30%, -0.1=-10%） |
| 阵法经验值 | 参考相同定位的阵法数值，与用户确认 |
| 坚韧类p1/p2 | 由战斗策划在设计层定义，数值策划照填 |

## 确认展示铁规

> ⚠️ 以下规则不可违反。

1. **必须用树形隶属格式**：查 `requirement_structures.md` 获取当前需求类型的结构模板，按父子关系缩进展示
2. **子行标来源**：每行标明参考行和关键修改
3. **叶子节点展示名字**：引用的已有实体必须查出名字展示，不能只给裸 ID
4. **数量必须可见**：子行数量（如掉落组个数）必须明确展示
5. **等用户确认后才写 filled.json**：树形展示 → 用户回复"确认" → 写文件
6. **每行必须有 `_ref_id`**：不支持从零建行。如果用户说不出参考谁，要求用户去源表找一个最接近的

## 输出格式

```json
{
  "requirement": "需求描述",
  "tables": {
    "表名": [{
      "_ref_id": "参考ID",
      "_note": "自然语言说明",
      "_overrides": { "字段": "值" }
    }]
  }
}
```
