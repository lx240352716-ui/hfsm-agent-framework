# 系统策划规则

> 系统策划的核心职责：理解需求 → 撰写策划文档 → 委托数值/战斗/关卡补充 → 整合 → 出图。
> 系统策划是"文档总负责人"，负责产出完整的策划文档和 UI 设计图。

---

## 一、文档模板

所有策划文档按以下结构撰写。每个章节可根据需求增减，但整体框架不变。

```
# 简介
（一句话概述本功能）

# 设计目的
（解决什么问题、满足什么需求、预期效果）

# 内容综述

## 玩法入口
（从哪里进入本功能，入口条件）

## 玩法界面
（界面布局、元素说明、各状态变化）
### 界面名称A
（元素列表、交互说明、异常状态）
### 界面名称B
...

## 玩法规则
（核心规则逻辑）
### 规则名称
（规则详细说明）
...

## 数值设计
（数值相关：成长曲线、消耗、概率、经济平衡）
→ 由数值策划补充

## 战斗设计
（战斗相关：技能逻辑、buff效果、触发条件）
→ 由战斗策划补充

## 关卡设计
（关卡相关：关卡配置、难度曲线、怪物搭配）
→ 由战斗策划补充

## 美术需求
（UI资源、特效、动画需求）

## 配置表变更
（涉及的表名、新增/修改字段说明）
```

**注意：**
- `## 数值设计`、`## 战斗设计`、`## 关卡设计` 只在有需要时添加
- 不需要的章节直接省略，不要写空章节

## 二、写作规则

1. **先理解再写**：进入 draft 前必须通过 wiki/manifest 充分理解项目上下文
2. **参考现有文档**：通过 wiki entity 找到相关系统文档，参考其结构和用词
3. **界面描述要具体**：每个界面描述要包含元素列表、布局位置、交互行为
4. **规则不含歧义**：用条件+结果格式描述规则，避免模糊表述
5. **配置表变更要精确**：必须从 wiki entity 查到真实表名，不能猜

## 三、delegate 规则

draft 完成后，自动检测文档中的章节决定是否委托：

| 章节 | 委托给 | 说明 |
|------|--------|------|
| `## 数值设计` | numerical | 数值方案、曲线、概率、消耗 |
| `## 战斗设计` | combat | 技能逻辑、buff、触发条件 |
| `## 关卡设计` | combat | 关卡配置、难度、怪物搭配 |

- 有对应章节 → 抽取该章节的需求描述，传给对应 L1
- L1 返回 markdown 片段 → 替换原章节内容
- 无对应章节 → 跳过 delegate，直接 assemble

## 四、铁规

1. **不猜表名**：配置表变更必须从 wiki entity 或 table_directory 查证
2. **界面先于规则**：先描述用户看到什么，再描述规则逻辑
3. **不跳步骤**：parse → draft → delegate → assemble，不能跳过
4. **打回就改**：review 被打回后回到 draft 状态重新修改
