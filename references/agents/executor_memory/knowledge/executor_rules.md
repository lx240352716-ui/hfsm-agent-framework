# 防坑铁规 (Executor Rules)

> S8私有。被QA打回或写坏文件后，将教训提炼为铁规 append 写入。动笔前必查，优先级高于一切外部模板。

## 条目 (Append-Only)

### 场景：新增行位置错误（追加到末尾）
- **踩坑记录**：新角色默认追加到表格末尾，但用户要求新行必须紧跟参考实体后面
- **防爆动作**：所有表的新增行必须插入到参考实体的下一行。**严禁追加到末尾**

### 场景：复制参考行后残留原角色名
- **踩坑记录**：从黑胡子复制整行到库赞后，OPDex 扩展字段(34处)和 HeroFashion 描述字段(3处)仍然写着"未来岛黑胡子"
- **防爆动作**：复制参考行完成后，必须对新行**全列扫描**，将所有包含原角色名的文本替换为新角色名

### 场景：大表写入（Buff 等 15000+ 行）
- **踩坑记录**：openpyxl 加载大表（load_workbook）会卡死，insert_rows 在大表上超时/损坏
- **防爆动作**：Excel 写入统一使用 COM 自动化（`win32com.client`），跟手动操作一样快，不受表大小限制。读取走 SQLite 索引

### 场景：FormationPos 等多行关联表结构误读
- **踩坑记录**：FormationPos 的 A 列是自增 ID 而非阵法 ID，B 列才是阵法 ID，C 列是占位 ID，每个阵法对应 5 行。误把 A 列当阵法 ID 只写了 1 行
- **防爆动作**：对不熟悉的表，必须在 S2a 阶段**先读取参考实体的完整行数据**确认列含义和每实体行数，再向用户确认不确定的字段（如占位ID）。**严禁盲猜列含义**

### 场景：BuffActive 百分比参数填成万分比
- **踩坑记录**：speedの15%填成1500(万分比)，实际应填0.15(小数)。SQLite查询现有数据确认格式为小数
- **防爆动作**：BuffActive.buff参数1 的百分比数值**一律用小数格式**（15%=0.15，20%=0.2），**不是万分比**。填表前先用SQLite查询同因子的现有条目确认数值量级

### 场景：UR装备套装遗漏表（来自 s6_equip_check）
- **踩坑记录**：UR装备套装涉及 11 张需新增行的表，极容易遗漏
- **防爆动作**：UR装备需逐表确认：
  - Equipment(4行) / _EquipRandProp(1行) / EquipUpgrade(16行) / EquipSuit(2行) / EquipBox(4行)
  - _EquipmentReIdentifyConfig(4行) / EquipmentHoleExpand(12行) / EquipResonance(8行)
  - FightBuff / _Buff / BuffActive（战斗表，行数由设计决定）
  - EquipRefine **按部位共享，不需新增**
  - BuffView **需前端确认，不适合程序推导**

### 场景：fill 阶段空字段盲猜导致数据错误
- **踩坑记录**：LLM 补值时遇到不确定的字段直接填 0 或随意猜测，导致写入 Excel 后数据错误
- **防爆动作**：fill 阶段不确定的字段**必须标 `uncertain: true`**，附 reason 说明原因。严禁沉默填 0、空字符串或盲猜。详见 `executor_fill_rules.md` 三重查询降级规则
