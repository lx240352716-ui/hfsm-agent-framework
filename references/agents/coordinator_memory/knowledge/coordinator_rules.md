# 主策划拆分规则

> 主策划的核心职责：理解需求 → 拆分模块 → 分给设计层（combat / numerical）。
> 主策划**不直接**分配给执行策划（executor 从设计产出自动推导）。
> 不确定 → 向用户确认 → 学到后写入 `coordinator_examples.md`。

---

## 一、设计层角色分工

| 角色 | 负责领域 | 关键词 |
|------|---------|--------|
| 战斗策划 | 技能逻辑、buff效果、触发条件 | buff、技能、被动、套装效果、阵法效果 |
| 数值策划 | 属性成长、数值曲线、概率、经济 | 数值、成长、曲线、概率、经验、属性 |

## 二、需求 → 是否需要战斗策划

| 需求类型 | 需要战斗策划 | 说明 |
|---------|:---:|------|
| 新角色(SSR) | 视情况 | 有新技能才需要，纯数值角色不需要 |
| 被动技能 | ✅ | 必须走战斗策划拆解 |
| UR装备 | ✅ | 套装效果buff + 增强buff |
| 新阵法 | ✅ | 有 FightBuff 配置 |
| 纯数值调整 | ❌ | 直接走数值策划 |

## 三、需求拆分模板

### 新角色
- 战斗策划: 技能、被动
- 数值策划: 角色属性、数值成长、抽卡

### UR装备
- 战斗策划: 套装效果buff、增强buff
- 数值策划: 属性、强化曲线

### 新阵法
- 战斗策划: 阵法buff效果
- 数值策划: 经验曲线、助阵人数

### 被动技能
- 战斗策划: buff逻辑、触发条件、效果值

### 纯数值调整
- 数值策划: 数值计算、曲线调整

## 四、铁规

1. **功能模块先确认**：任何新系统/新模块，必须先向用户列出功能模块清单并确认，再进入设计
2. **不确定就问**：不确定的需求必须向用户逐条确认，不要猜
3. **不派执行**：主策划只分给 combat 和 numerical，执行策划由上游产出自动驱动
4. **案例只存不写**：中间状态（parse/split/dispatch）只往 `pending_examples.json` 追加，禁止直接写 `examples.md`。只有 L0 review 阶段统一从 pending 提交到 examples.md

## 五、复盘规则

```
1. 收集本轮所有问题(错误、用户确认项)
2. 去重: 是否该问题曾出现过?
   - 首犯 → 写入本文件踩坑记录
   - 重犯 → 升级为铁规
3. 发布复盘报告
```

## 六、踩坑记录

| 错误 | 教训 |
|------|------|
| 跳过模块确认直接到表级别 | 铁规：功能模块先确认 |
| 以为 EquipRefine 需要新增 | 按部位共享机制，新套装不需新增 |
| 以为装备本体要注册 Item 表 | 装备本体由 Equipment 管理，只有装备任选箱需要 Item 注册 |
