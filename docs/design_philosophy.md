# 设计思路：为什么选 FSM+Hook

## 核心问题

LLM 在生产环境中会产生幻觉。当输出直接影响线上系统时（如游戏配表、数据库变更、API 配置），一个错误就是一个事故。

## 三种方案对比

### Pipeline（链式调用）

```
LLM → LLM → LLM → LLM → 输出
      ↑ 幻觉逐级放大
```

- 每步的输出是下步的输入
- 上游幻觉传导到下游，下游无法纠正
- 调试困难：错误发生在哪一步？

### Agent-Chain（自主 Agent）

```
LLM 决定做什么 → LLM 执行 → LLM 判断是否完成
      ↑ 不可预测
```

- LLM 自主决定下一步行动
- 灵活但不可控：可能跳步、重复、死循环
- 安全关键场景不可接受

### FSM+Hook（本框架）

```
状态机 → [状态A] → Hook校验✓ → [状态B] → Hook校验✓ → 输出
          ↑ 幻觉被限制在单步内
```

- 状态机决定流程，Hook 约束每步能做什么
- LLM 只在 Hook 划定的边界内工作
- 每步输出经过代码校验才能进入下一步

## 四个关键选型

### 1. 状态机代替 Pipeline

状态机把每一步的输入、输出、能力边界都锁死。幻觉被限制在单步内，不会扩散。

### 2. Hook 函数代替 LLM 自主决策

`on_enter_*` 限定输入范围（加载什么知识、读什么数据），`on_exit_*` 校验输出（格式、合法性、完整性）。LLM 在 Hook 划定的边界内工作，不能越权。

### 3. JSON Handoff 代替 LLM 传话

Agent 间数据通过结构化 JSON 直接交接。LLM 不参与数据中转，消除转述过程中的信息失真。

### 4. 文件记忆代替 LLM 记忆

Agent 的知识从 Markdown 文件按需加载，不依赖 LLM 的对话上下文。文件是持久的、可版本管理的，策划也能直接编辑来修正 AI 行为。

## 一句话总结

> Pipeline 假设 AI 是对的，FSM 假设 AI 会出错 — 然后用工程手段把错误关在笼子里。
