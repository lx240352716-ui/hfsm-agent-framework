## 2. 机制级业务知识吸收
- **精炼表机制(EquipRefine)**：`EquipRefine.xlsx` 不是挂载在 `equipId` 上，而是挂载在 `装备部位` (1=武器,2=防具,3=戒指,4=项链) 上。也就是所有套装的精炼成功率与消耗是全游戏共享一套。**结论：新增套装无需在精度表新增数据**。
- **动态概率梯度实现(BuffOdds + Grade)**:
  - 装备强化（1-4级）在战斗底层的传入就是 `grade=1,2,3,4`
  - 如果要实现随着强化等级上升，概率也上升（如武器威压特效 50%->80%），通过复用同一个 `FightBuff` 和 `_Buff`，并将其 `oddsId` 指向底表 `BuffOdds` 中一组针对 `grade=1,2,3,4` 分别配置不同概率的行。不需要配四个FightBuff。
  - 用户提示的 *"找有老的10%-100%的用那个参考他去填写"* 指的就是这个通过grade区分概率梯度的做法。

## 3. 表格定位知识更新
- 装备系统全套闭环涉及12个核心模块，共计13张可能需要新增行的表，包含：
  - **静态类**：Equipment, EquipBox
  - **强化消耗类**：EquipUpgrade, EquipmentHoleExpand, _EquipmentReIdentifyConfig, EquipResonance
  - **套装组装类**：EquipSuit, _EquipRandProp
  - **战斗黑盒类**：FightBuff, _Buff, BuffActive, BuffOdds, _BuffCondition, BuffView
- 注意：**BuffView** 是需要专门配置图标显示的表，需前端参与或纯手动填，不适合完全程序推导，交付时需让用户确认。
