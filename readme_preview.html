GameTable Copilot

用自然语言描述需求，AI 自动生成可导入的 Excel 配置表。

基于分层状态机 (HFSM) 驱动的多 Agent 系统，用于游戏配表自动化。

解决什么问题

游戏策划需要手动维护大量 Excel 配表（技能、Buff、掉落、商店...），这个过程：

痛点 具体表现
效率低 一个被动技能可能涉及 5-8 张表、几十个字段，手动查找填写耗时数小时
易出错 字段名中英混杂，ID 关联复杂，漏填一个引用就全盘报错
知识壁垒高 新人要理解上千张表的关系和填写规则，上手周期长
难以验证 填完不知道对不对，要等客户端跑起来才能发现问题

本项目让策划用自然语言描述需求，AI 自动完成 理解→拆解→定位表→查数据→填值→写入→校验 的全流程。

核心设计思路

配表的输出直接进客户端，一个错误的 ID 就是一个线上问题。所以整个系统的前提是：LLM 会出错，工程侧必须兜底。

基于这个前提，系统在四个环节做了对应的选型：

环节 常规做法 我们的做法 原因
流程编排 Pipeline / Agent-Chain 分层状态机 (HFSM) Pipeline 中上游的幻觉会传导到下游，逐步放大。状态机在每一步设置校验关卡，错误不会扩散到下一步
步骤执行 LLM 自主决定下一步做什么 Hook 函数约束每步能做什么 on_enter_* 限定输入范围，on_exit_* 校验输出。LLM 在 Hook 划定的边界内工作，不能越权
Agent 间通信 LLM 总结上一步结果后传给下一步 结构化 JSON 直接交接 LLM 转述会丢信息、改数值、编造字段。JSON 是确定性的，下游拿到的就是上游写入的
知识管理 依赖 LLM 对话记忆 每次从 Markdown 文件按需加载 对话上下文有窗口限制，长了会丢。文件是持久的，可版本管理，策划能直接编辑来修正 AI 行为

技术选型

组件 选型 选择原因
流程编排 自研 HFSM 比 Pipeline 更安全，状态可持久化和回滚
Agent 通信 JSON Handoff 结构化传递，避免 LLM 转述失真
步骤实现 Hook 函数 约束 LLM 能力边界，可插拔扩展
知识管理 Markdown 文件 策划可直接编辑，不依赖 LLM 记忆
数据索引 SQLite 1000+ 张表毫秒级跨表查询
Excel 读写 openpyxl 直接操作源文件
